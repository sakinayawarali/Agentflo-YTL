# Auth + Onboarding (Invoice Verification) Handover

This document describes the authentication gatekeeper and the invoice-based onboarding flow for unauthenticated users.

## Overview

The WhatsApp message handler enforces authentication before invoking the agent. If the user is not in the customer database, the system switches to an onboarding flow that requests a last‑month invoice image and verifies it via the backend Invoice Verification API. Successful verification marks the user as authenticated and allows normal agent flow.

## Key Files

- `agents/helpers/adk_helper.py`: Auth gatekeeper and onboarding prompt.
- `agents/helpers/route_handlers.py`: Invoice image interception + verification pipeline.
- `agents/helpers/image_helper.py`: Gemini invoice extraction (structured JSON).
- `agents/helpers/session_helper.py`: Auth + onboarding state persistence.
- `agents/tools/api_tools.py`: `search_customer_by_phone` + `verify_invoice`.

## Auth Gatekeeper Flow (Text / Any Message)

Entry point: `ADKHelper.handle_message_async` in `agents/helpers/adk_helper.py`.

1) Fetch `is_authenticated` from Firestore via `SessionStore.get_auth_status`.
2) If an `external_user_id` is already stored, mark authenticated.
3) If not authenticated:
   - If `onboarding_status == "awaiting_invoice"`, return the onboarding prompt immediately (no API calls).
   - Otherwise call `search_customer_by_phone(wa_user_id)`:
     - If success: set `is_authenticated = True` and clear onboarding status.
     - If failure: set onboarding status to `awaiting_invoice` and return the invoice prompt.

Prompt message (Roman Urdu) is generated by `_get_onboarding_invoice_prompt`.

## Onboarding Flow (Invoice Image)

Entry point: image handling in `agents/helpers/route_handlers.py`.

When a message of type `image` arrives:

1) If user is **not authenticated**, the image is treated as an onboarding invoice.
2) Image is downloaded:
   - Meta: `WhatsAppImageHelper.get_invoice_from_image_structured_output(media_id)`
   - Twilio: `_download_twilio_media` → `get_invoice_from_image_path(img_path)`
3) Gemini extracts structured JSON using the invoice prompt + schema.
4) Payload is normalized (dates, phone, arrays).
5) Step 1 (first invoice):
   - Store first invoice fields in Firestore (`onboarding_invoice_1`)
   - Set `onboarding_status = "awaiting_invoice_2"`
   - Reply: ask for the second invoice
6) Step 2 (second invoice):
   - Combine first + second invoice into one payload
   - Call `verify_invoice(...)`
7) On success:
   - `is_authenticated = True`
   - `onboarding_status = None`
   - Store `external_user_id` if present in API response
   - Store `store_code` metadata in the user doc (best effort)
   - Reply: “invoice verified, you can order now”
8) On failure:
   - `onboarding_status = "awaiting_invoice_2"`
   - Reply: ask to resend the second invoice photo

## Data Model (Firestore)

User path: `tenants/{TENANT_ID}/users/{wa_user_id}`

Session meta doc:
`tenants/{TENANT_ID}/users/{wa_user_id}/sessions_meta/meta`

Relevant fields:
- `is_authenticated` (bool)
- `onboarding_status` (string; e.g., `awaiting_invoice_1`, `awaiting_invoice_2`)
- `onboarding_status_at` (float timestamp)
- `onboarding_status_reason` (string)
- `onboarding_attempts` (int)
- `onboarding_invoice_1` (dict; first invoice fields)
- `external_user_id` (string, in user root doc)
- `store_code`, `storecode` (string, in user root doc)

## External APIs

1) Customer lookup:
`search_customer_by_phone(wa_user_id)` in `agents/tools/api_tools.py`

2) Invoice verification:
`verify_invoice(...)` in `agents/tools/api_tools.py`

Expected verify payload:
```
{
  "tenant_id": "<TENANT_ID>",
  "mobile_number": "92XXXXXXXXXX", // always from WhatsApp sender number
  "invoice_numbers": ["D0663INV11816"],
  "store_codes": ["N00000313606"],
  "delivery_dates": ["2026-01-07"]
}
```

Verification success is inferred from:
- `success == true`, or
- `status` in {`success`, `verified`, `valid`, `ok`}

If the response contains a canonical user id in `data` / `customer`:
- `user_id`, `customer_id`, or `id` is saved as `external_user_id`.

## Gemini Invoice Extraction

Implemented in `agents/helpers/image_helper.py`:

- Schema: `InvoiceExtraction`
- Prompt enforces extraction rules (invoice number, store code, delivery date, CNIC, mobile number).
- `tenant_id` is taken from the `TENANT_ID` environment variable for the onboarding flow only.
- Model override: `INVOICE_VISION_MODEL` (optional).

## Environment Variables

Required for onboarding:
- `TENANT_ID` (used for invoice verification payload and Firestore tenant path)
- `API_JWT_TOKEN` (auth for backend APIs)
- `INVOICE_VERIFY_URL` or `INVOICE_VERIFY_BASE_URL`
- `WHATSAPP_ACCESS_TOKEN` (for Meta image fetch)

Optional:
- `INVOICE_VISION_MODEL` (override Gemini model for invoice extraction)
- `WHATSAPP_TRANSPORT=twilio` (Twilio flow uses media download)

## Failure Modes + Recovery

- No customer found in API → prompt invoice upload.
- Invoice extraction fails → ask for a clearer image and reset status to `awaiting_invoice`.
- Verification fails → ask for a clearer invoice and reset status to `awaiting_invoice`.
- If `TENANT_ID` is unset, onboarding verification will send an empty tenant id (intended per current requirement).

## Operational Notes

- Auth gatekeeper runs for **all** messages before agent invocation.
- Invoice onboarding is only triggered for **image** messages when the user is not authenticated.
- Onboarding state is persisted, so the prompt is shown consistently across sessions.
